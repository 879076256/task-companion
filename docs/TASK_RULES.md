# 任务规则

## 1. 原始任务与正式任务

原始任务继续使用 Markdown Tasks。只有未完成、未取消且包含 `⏫`、`🔼`、`🔽`、`⏬` 或 `🔁` 的任务才进入 Task Companion。

以下内容必须排除：

- `[x]`、`[X]` 已完成任务；
- `[-]` 已取消任务；
- 文本中明确包含 `cancelled`、`canceled` 或“取消”的任务；
- 没有优先级或循环标记的普通复选框。

## 2. 今日待办

任务未完成、未取消且属于正式任务，并符合以下任一条件时进入今日待办：

1. 同时有 Start（🛫）和 Due（📅），且 `Start ≤ 今天 ≤ Due`；
2. Scheduled（⏳）是今天或早于今天；
3. Due 是今天；
4. Due 早于今天且仍未完成，标记为“逾期”；
5. 只有 Start、没有 Due，且 `Start ≤ 今天`；
6. 只有未来 Due、没有 Start，在截止前不进入；
7. 没有任何日期，不进入今日待办。

Scheduled 已到期和已逾期 Due 优先成立，即使其他日期组合不一致，也不能隐藏需要处理的任务。

## 3. 重点、日常与待推进任务

- 所有活动的 `⏫` 任务都是重点任务，不受日期限制。
- 同时属于今日与重点的任务只显示一次，分类为“今日待办＋重点任务”。
- 日常候选是活动的 `🔁` 任务。存在 Start、Scheduled 或 Due 时，最早显式日期到达后才显示；没有显式日期时，根据同名任务最近的 `✅` 完成日期和 recurrence 规则计算下一次可用日期。
- recurrence 支持 weekday、具体星期以及按日/周/月/年的整数间隔；不能解析规则时，从最近完成日的次日重新可用。
- `🛏️ 早起`、`📖 reading`、`🏃🏻 运动` 三类打卡只接受位于今日日记 `Calendar/Journal/Daily/YYYY-MM-DD.md` 的任务。
- 所有带 recurrence 的日常任务都属于提醒型任务：保留在“日常任务”提醒，但从当前任务、快速推进和完成任务等所有选择入口排除。是否排除只依据 `hasRecurrence`，非循环的今日、重点和待推进任务不受影响。
- 待推进任务是未进入今日、不是循环、也不是 `⏫` 的其余活动正式任务。

## 4. 稳定任务 ID

首次安全扫描正式任务时，在任务行末尾追加六位小写十六进制块 ID：

```markdown
^tc-a7f3d2
```

规则：

- 已有有效 `^tc-xxxxxx` 保持不变；
- 已有其他块 ID 时保持原文并报告冲突；
- 同一扫描中 ID 必须唯一；重复 ID 的后续任务不进入选择结果；
- 写入前比较完整文件和目标原行；任何冲突或写入失败都保持原文；
- 不写入番茄图标、番茄次数或其他显示字段。

## 5. 扫描和显示

- 只扫描当前用户明确打开任务选择器的 Vault 中的 Markdown 文件。
- 开发与验收只允许仓库内 `test-vault/`。
- 扫描失败显示数量汇总，不在日志或通知中暴露任务正文。
- 单个文件失败不阻断其他文件。
- 选择器显示任务正文、分类、日期和 Vault 相对来源路径，并支持显式刷新。

## 6. Phase 7 嵌入式任务组件

- `today` 只显示分类为 today 或 today-important 的任务。
- `important` 只显示分类为 important 或 today-important 的任务。
- `daily` 显示所有活动循环任务，包括同时属于今日或重点的循环任务。
- 同一组件内按稳定 taskId 保持唯一；任务可以合理地同时出现在不同组件中。
- `current` 按当前绑定的稳定 taskId 从全部活动正式任务中读取，不因日期跨日退出当前任务组件。
- 组件首次渲染时扫描；today、important、daily 和 review 保留显式刷新。`current` 不显示刷新按钮，依靠当前 taskId、执行目标、计时、会话或子任务变化自动同步；直接编辑来源 Markdown 后可通过重新打开页面或后续内部状态变化重读。同一主页同时加载的任务组件共享一次进行中的扫描，避免安全写入竞争。
- `current` 必须明确区分“母任务直接推进”和具体活动子任务，并展示实时计时状态。

## 7. Phase 7.4 操作、完成与复盘资格

- 今日、重点和日常组件的任务标题进入执行准备；标题前的未选复选框触发安全完成，标题旁只保留 `🔗` 来源入口。
- 来源跳转以扫描行号为定位提示，优先把编辑器光标和视口定位到任务行；无法精确定位时仍必须打开来源页。
- 当前任务选择器以今日待办、重点任务、日常任务为一级视觉标题，以母任务为二级标题；活动子任务只在用户展开母任务后作为三级标题显示，母任务完成同样使用标题前复选框。
- `current` 只显示母任务和当前选中的活动子任务，不显示执行表单中的自由文字 `nextAction`。
- `current` 紫色目标框中的母任务标题与今日/重点/日常任务标题使用同一执行准备入口，可重新选择母任务直接推进或任一活动子任务；选择子任务后目标框自动显示母任务、子任务两行。
- 执行准备必须在单一窗口中同时显示母任务、全部子任务和底部新增框；活动子任务可选择为执行目标，已完成或已取消子任务只保留管理入口。每个子任务的选择按钮后提供更多菜单，不得再通过“编辑任务拆解”打开第二个窗口。
- 只有用户明确完成母任务或子任务才生成/恢复 pending 复盘；结束或自然完成一次专注计时只保存 ExecutionSession，不改变任务完成态，也不弹出执行收尾表单。25 分钟自然结束另显示计时完成提醒并进入自动休息，该提醒不等同于任务完成或复盘。
- 当前任务卡的完成按钮完成当前执行目标：已选活动子任务时完成该子任务，否则完成母任务；成功后才弹出“本次执行已结束”，字段为“本次完成了什么 / 下一步是什么 / 注意事项（可选）”。
- `current` 显示活动子任务时，子任务行也提供完成按钮；该按钮与母任务完成按钮纵向对齐。母任务按钮只完成母任务并执行未完成子任务分流，子任务按钮只完成当前选中的活动子任务；两者成功后都进入执行收尾表单。
- pending 复盘项使用已选复选框；用户取消勾选时，母任务恢复为 Markdown `[ ]`，子任务恢复为 `active`，原 pending 事件保留但因不再符合完成资格而隐藏。再次完成时恢复同一待复盘项，避免重复。
- 已经保存的 completed 复盘属于历史记录，不因后续返工而删除；用户明确永久删除该子任务时例外。

## 8. 子任务取消与永久删除

- “取消子任务”只把活动子任务改为 `cancelled`：不再允许选择为执行目标，但保留子任务、全部执行会话和复盘历史，并可恢复为进行中。
- “删除子任务”对 `active`、`cancelled`、`completed` 三种状态都开放，不要求先取消。
- 删除是不可撤销的目标级清理：移除该 `taskId + subtaskId` 的全部子任务事件引用、ExecutionSession、pending 会话、子任务 ReviewEvent、pending 复盘和对应可读复盘 Markdown；其他母任务、兄弟子任务和无法解析的日志行必须保持不变。
- 若删除目标正在运行或暂停计时，先立即重置计时且不得生成结束会话，再清除当前子任务绑定。
- 清理顺序先处理会话与复盘，最后移除子任务档案；任一步失败都显示错误并保留可见子任务供用户重试，不把部分失败报告为删除成功。

## 10. 倒计时提醒与番茄循环

- 只有自然归零触发声音、完成弹窗和自动阶段切换；暂停、重置和提前结束不触发。
- 普通 25 分钟专注按完成次数累计轮次；每第 3 次自然完成后准备 15 分钟休息，其余准备 5 分钟休息。
- 专注或休息自然结束后只进入下一阶段待开始状态，用户再次点击开始后才运行倒计时；休息阶段不生成任务 ExecutionSession，不计入投入和复盘。
- 后台系统通知是尽力而为能力；权限拒绝、勿扰模式或平台失败不得影响本地弹窗、声音、持久化或下一阶段准备。

## 9. Phase 8 经验模板与今日完成

- 只有母任务复盘成功保存后才询问是否沉淀经验；完成任务本身不自动创建模板，子任务复盘不创建母任务模板。
- 用户可新建模板、更新一个已有模板或暂不沉淀；任何选择都不得影响已经保存的复盘。
- 模板子任务只收录非取消步骤并保持顺序；历史统计按独立复盘聚合，同一 reviewId 不得重复计数。
- 执行准备按任务标题相似度和更新时间显示模板建议；采用前只读，采用后一次性追加缺失标题，已有同名子任务保持不变。
- 模板采用生成的子任务 origin 为 `template`，仍是普通一层子任务，可由用户编辑、完成、取消或删除。
- 复盘主页“今日已完成”使用 ReviewEvent 的 `completedAt` 按当前本地日期筛选，而不是按复盘保存日期筛选。
- 同一 `taskId + subtaskId` 当天出现多个复盘快照或重复完成记录时，今日队列只显示最新一项；母任务使用空 subtaskId，母任务与其子任务仍是不同目标。
- “今日已完成”指标保持与“待复盘”同形；点击进入今日队列，命令面板的完整复盘队列不受日期过滤。

## 10. Phase 10 测试主页四类提醒

- 插件一次派生 `daily`、`today`、`important`、`pending` 四组主页快照；嵌入组件和 `Home-Test` API 使用同一纯分组规则。
- `tasks.homeReminders(date)` 使用只读扫描：不追加稳定 ID、不调用 Vault process、不修改任务来源笔记；返回 Vault 相对路径和一基行号，不返回绝对路径。
- `Home/Home-Test.md` 只把原 Dataview 扫描、筛选和排序替换为插件快照；既有 DOM、CSS 类、标签页、指标、完成/编辑/来源处理和排版代码保持不变。
